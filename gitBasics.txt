Flujo diario de cada persona
Cada día que trabajas, el ciclo es siempre el mismo:
bash# 1. Asegurarte de tener lo último de tu rama
git pull origin feature/mapa-bicis

# 2. Trabajar en tus ficheros...

# 3. Ver qué has cambiado
git status
git diff

# 4. Añadir y commitear
git add .
git commit -m "...."

# 5. Subir
git push origin feature/mapa-bicis  [u otra rama]
Mensajes de commit útiles:
feat: nueva funcionalidad
fix: corrección de bug
style: cambios de CSS
refactor: reorganización de código
wip: trabajo en progreso (commit provisional)

Mantenerse sincronizado con develop
Cuando la Persona A sube cambios al HTML base que os afectan a todos, tenéis que traeros esos cambios a vuestra rama para no quedaros desfasados:
bash# Estando en tu rama feature
git checkout feature/mapa-bicis

# Traer los cambios nuevos de develop a tu rama
git fetch origin
git merge origin/develop
Si hay conflictos Git os avisará. Normalmente no los habrá si cada uno toca sus propios ficheros.


El merge final a develop y luego a main
Fase 1: cada persona abre un Pull Request
Cuando una persona termina su parte, va a GitHub y hace esto:
GitHub → pestaña "Pull requests" → New pull request
  base: develop
  compare: feature/mapa-bicis
  Título: "Fase I: mapa de bicis con datos estáticos"
  → Create pull request 

Opción A — Por GitHub (recomendada para trabajo en equipo)
Vas a GitHub → Pull requests → New pull request y seleccionas:

base: develop
compare: feature/mapa-bicis
Luego en tu máquina actualizas develop:
bashgit checkout develop
git pull origin develop

Opción B — Por consola (si trabajáis solos o de acuerdo)
bashgit checkout develop
git pull origin develop           # asegurarte de tener develop al día
git merge feature/mapa-bicis      # traer tus cambios a develop
git push origin develop           # subir develop actualizado
